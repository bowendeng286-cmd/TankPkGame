# AGENTS.md

This file provides guidance to agents when working with code in this repository.

## 项目概述
纯前端 HTML5 坦克对战游戏（Tank Trouble 克隆），无构建系统，直接在浏览器中运行。

## 运行方式
直接用浏览器打开 `index.html`，无需构建或安装依赖。

## 完成通知规则
- 任务真正完成后，在发送最终回复前，应触发完成通知
- 完成通知需要播放 3 声提示音，并弹出提示框告知用户任务已经完成
- 如果任务被阻塞、只完成部分内容、或仅进行分析说明，则不要触发完成通知

## AI 协作开发工作规范（强制）

以下规范强制纳入本仓库的 `AGENTS.md` 执行规则中，AI 在本仓库工作时必须遵守；如与其他低优先级说明冲突，以本节要求为准。

### 目标
- AI 的任务是协助用户完成编程项目开发。
- 鉴于用户编程基础有限，所有改动应尽量清楚、稳妥、可验证。
- AI 应将本节视为仓库内的高优先级执行守则。

### 协作原则
1. 每次开始修改前，先说明准备修改的内容、会影响哪些文件、以及可能存在的风险。
2. 尽量做小改动，不要一次性重写过多代码。
3. 如果存在多种实现方式，优先选择最简单、最稳妥、最容易维护的方案。
4. 不要默认用户能直接看懂复杂代码；解释关键点时要尽量使用用户容易理解的方式。

### 代码完成后的要求
1. 每次写完代码后，AI 必须先自行审查一遍，再交给用户验证。
2. 自查重点包括：
   - 逻辑是否正确
   - 有没有明显语法错误
   - 有没有遗漏边界情况
   - 有没有影响到其他文件或功能
   - 有没有不必要的复杂写法
3. 如果发现改动中存在疑点，AI 必须先主动说明，不能直接把问题留给用户。
4. 完成自查后，再让用户实际运行程序或测试，作为最终审查。
5. 每次任务真正完成后，AI 必须在 `AGENTS.md` 中补充记录本轮开发经验，不能省略。
6. 记录内容必须包括本轮开发中踩到的坑，既包括编码时遇到的坑，也包括与用户沟通和理解需求时遇到的坑。
7. 与用户沟通相关的坑包括但不限于：用户描述与 AI 理解不一致、同一组件或功能的称呼不同导致理解偏差、需求边界表达不完整、用户默认前提未明说等。
8. 这些记录应尽量具体、详细、可复用，目标是在后续开发时减少重复犯错并提升开发效率。
9. 除了坑点记录，AI 还必须补充本轮开发的内容摘要和进度状态，让后续继续开发时能快速理解当前阶段。
10. 如果本轮没有明显坑点，也要明确写明“本轮未发现明显坑点”，并记录本轮完成内容与当前进度。

### 用户负责的部分
1. 用户主要负责按照说明运行程序、点击界面、执行实际测试。
2. 如果用户反馈报错或行为异常，AI 需要根据反馈继续修正。
3. 由于用户编程基础有限，AI 应尽量明确告知：该看哪里、该运行什么、预期结果是什么。

### Git 工作流
1. 不要直接在 `main` 分支上随意改代码。
2. 每个任务尽量单独创建一个分支，例如 `feature/login-form`、`fix/api-timeout`、`refactor/config-loader`。
3. 修改前先检查当前状态，确保清楚有哪些文件会被影响。
4. 修改后先查看 `git diff`，确认改动符合预期。
5. 每次只提交一个清晰的小改动，不要把很多无关内容混在一起。
6. 提交信息要说明改动目的，例如 `fix login validation`、`add config loader`、`refactor error handling`。
7. 在用户确认可以之后，再合并到主分支。
8. 本仓库的 git 和分支管理默认尽量由 AI 全权负责，包括建分支、提交、合并、整理改动和处理常规仓库操作。
9. 用户不擅长管理仓库，也不会主动查看代码和仓库改动；AI 应将仓库管理工作视为自己的职责范围。
10. 用户的主要职责是在 AI 完成一次改动并说明后，反馈“没问题”或“还需要改”。
11. 在用户明确确认之后，后续的合并、整理和其他 git 操作由 AI 负责，不需要用户自行处理。
12. AI 应主动负责仓库节奏控制，不要把仓库维护、分支整理、合并判断等事务推给用户。
13. 如果某个 git 操作确实需要用户参与，AI 必须用最简单、最明确的话告诉用户只需要做什么，不要让用户自己判断仓库细节。
14. 每次完成一轮任务后，AI 必须明确告诉用户现在需要进行测试，并说明用户应看哪里、怎么测、预期结果是什么。
15. 只有在用户测试后明确表示“可以”“没问题”或同等确认后，AI 才能执行合并操作；这里的合并既包括合并到 `main`，也包括合并到其他支线。
16. 即使当前已经在某个支线上工作，只要开始新的功能开发或 Debug 任务，AI 也必须再新建一个新分支，并在该新分支上工作，不能直接在原支线上继续叠加新任务。

### 推荐的实际流程
1. 先确认任务目标。
2. 为本轮任务创建专用分支；如果当前已经在某个支线上，但要开始新的功能开发或 Debug，也要继续新建分支，而不是直接沿用原分支。
3. 开始修改代码。
4. AI 自行审查修改。
5. 在 `AGENTS.md` 中记录本轮开发内容、进度和踩坑经验。
6. 明确告诉用户如何运行和验证，并直接说明现在需要用户进行测试。
7. 用户实际运行并反馈结果。
8. AI 根据反馈继续修正。
9. 用户明确确认“可以”之后，再由 AI 负责提交、合并到 `main` 或合并到其他目标分支。

### 需要避免的事情
1. 不要一次性改太多文件。
2. 不要在没有解释的情况下做大范围重构。
3. 不要跳过自查直接让用户测试。
4. 不要假设用户能自动发现问题，应尽量提前指出风险和验证方法。
5. 不要在任务完成后忘记把本轮踩坑经验、开发内容和进度写入 `AGENTS.md`。
6. 不要只写笼统结论，坑点记录必须尽量细，避免后续无法复用。

### 简短原则
先想清楚，再改代码。先自查，再让用户运行。先用分支管理改动，再通过 `git diff` 和小提交确认质量。

再次强调：这个仓库以 AI 主导管理为默认协作方式，用户主要负责确认结果，不负责仓库运维。

## 开发记录维护规则（强制）

AI 在每次任务真正完成后，必须继续维护 `AGENTS.md` 中的开发记录，作为后续开发的重要上下文来源。

### 必须记录的内容
1. 本轮任务做了什么，改了哪些文件，当前做到哪一步。
2. 本轮开发进度，例如：已完成、部分完成、待用户验证、等待新需求、等待进一步修正。
3. 本轮踩到的编码坑，例如：变量作用域误判、脚本加载顺序问题、全局状态依赖遗漏、样式或交互联动理解错误、调试中发现的隐藏约束。
4. 本轮踩到的沟通坑，例如：用户说法与代码中的实际命名不同、用户描述偏口语导致功能边界理解偏差、AI 自己对用户意图做了错误推断、同一个概念在不同上下文下含义不同。
5. 后续开发时需要特别注意的事项，避免重复踩坑。

### 记录要求
1. 记录要尽量详细，不能只写过于笼统的一句话。
2. 记录必须可供未来直接复用，让后续 AI 能快速理解历史问题和当前状态。
3. 每条开发记录的标题时间必须精确到分钟，格式应类似 `2026-04-18 15:10`，不能只写日期不写时间。
4. 如果某个坑是因为 AI 自己理解失误造成的，也必须明确写出误解点和修正后的正确理解。
5. 如果本轮用户表达和 AI 理解存在偏差，必须把双方各自的表达方式一并记下，帮助后续对齐语义。
6. 如果本轮没有新增坑点，也要记录本轮未出现明显新坑，并补充本轮开发内容和进度。

### 目的
- 通过持续记录开发内容、进度和坑点，为后续开发提供稳定上下文。
- 降低 AI 重复犯错的概率。
- 降低因为命名差异、口语描述差异、需求理解偏差带来的沟通成本。
- 提升后续任务的开发效率和协作准确性。

### 推荐记录模板
后续每次任务真正完成后，AI 应优先按以下模板追加记录，避免遗漏关键上下文：

```markdown
## 开发记录

### [日期 时间（精确到分钟）] 任务标题
- 本轮开发内容：
  - 做了什么
  - 修改了哪些文件
  - 关键改动点是什么
- 当前进度：已完成 / 待用户验证 / 部分完成 / 等待进一步修正 / 等待新需求
- 编码坑点：
  - 没有则写：本轮未发现明显编码坑点
  - 有的话尽量写清楚触发条件、错误理解、修正方式、后续如何避免
- 沟通坑点：
  - 没有则写：本轮未发现明显沟通坑点
  - 有的话写清楚用户原话或原始表达、AI 最初如何理解、后来如何确认出正确理解
- 后续注意事项：
  - 下次继续开发时要优先注意什么
  - 哪些命名、边界、前提条件容易再次误解
```

AI 追加记录时应尽量保持格式稳定，便于后续快速查阅。

### [2026-04-18] 优化中等和困难 AI 的威胁预测开销
- 本轮开发内容：
  - 在 `js/ai.js` 中优化了 AI 威胁评估流程，重点处理“中等/困难难度更卡”时每帧重复做高成本子弹轨迹预测的问题。
  - 给子弹未来轨迹预测增加了共享缓存：同一帧内多台 AI 面对同一颗子弹时，如果子弹位置、速度、生命周期、反弹次数和预测时长都没变化，就直接复用已有轨迹结果，不再重复调用 `traceBulletSegments()`。
  - 增加了一个安全的距离预筛：如果按当前子弹速度和预测时长计算，理论上都不可能在本次预测窗口内接近当前坦克，就直接跳过后续完整轨迹分析，减少中高难 AI 在子弹较多时的无效计算。
  - 本轮修改文件为 `js/ai.js` 和 `AGENTS.md`。
- 当前进度：待用户验证
- 编码坑点：
  - 这次的性能问题不是单个 AI 内部某段代码“特别慢”这么简单，更大的开销来自“多台 AI 对同一颗子弹重复做同样的未来轨迹推演”。如果只盯着某个函数本身，很容易漏掉这种跨 AI 重复计算的成本来源。后续排查性能问题时，要优先看是否存在“对象级重复运算”。
  - 这类优化不能靠粗暴降低 AI 思考频率，否则虽然会更省算力，但很容易让困难难度的闪避和反应明显变笨。本轮因此优先选了“缓存已有结果”和“只跳过理论上不可能命中的远距离子弹”两种更稳妥的做法。
- 沟通坑点：
  - 用户原话是“我觉得坦克AI的中等和困难电脑有点卡，需要优化一下算法”。这里的“卡”有两种常见含义：一种是帧率卡、算法开销大；另一种是 AI 行为卡顿、决策发愣。
  - 本轮结合“需要优化一下算法”的说法，先按“性能开销偏大”理解并处理。如果用户后续反馈不是更流畅，而是“AI 还是会犹豫、走位发呆”，则应改按“行为决策节奏”方向继续查，不要把两种“卡”混为一谈。
- 后续注意事项：
  - 后续若继续优化 AI 性能，优先检查 `ai.js` 中的威胁预测、反弹射击搜索和闪避规划是否还能继续复用结果，而不是先削弱难度参数。
  - 用户验证时应重点观察“中等/困难 AI 对战时整体是否更流畅”，同时确认 AI 仍然会正常躲子弹、瞄准和开火，避免只优化了性能却悄悄削弱行为质量。

## 核心架构特点

### 1. 脚本加载顺序严格依赖（index.html）
**必须按以下顺序加载**，否则会出现未定义错误：
```
constants.js → theme.js → i18n.js → utils.js → state.js → controlsSettings.js 
→ input.js → maze.js → tank.js → bullet.js → particles.js → renderer.js 
→ physics.js → menu.js → controlsConfig.js → ai.js → main.js
```

### 2. 全局变量系统
- 所有常量在 `constants.js` 中定义为全局变量（无 `const`/`let`/`var` 前缀）
- 视口变换变量 `VIEWPORT_SCALE`、`VIEWPORT_OFFSET_X/Y` 使用 `var` 声明（需运行时修改）
- 游戏状态在 `main.js` 中声明为全局 `var`（`maze`、`tanks`、`bullets` 等）

### 3. 物理引擎的独特设计
**关键：输入速度与碰撞速度分离**
- 坦克的 `vx/vy/omega` 包含输入分量和碰撞残余分量
- 使用 `_lastInputVx/Vy/_lastInputOmega` 记录上一帧输入，用于提取碰撞残余
- 碰撞残余以 0.3 系数衰减保留，让坦克旋转时自然滑离墙壁
- **修改物理代码时必须保持这个分离机制**，否则会导致坦克卡墙或无法旋转

### 4. 炮管防穿墙的预测式系统
- `wouldBarrelCollide()` 纯检测函数，不修改任何状态
- `findMaxSafeOmega()` 使用二分搜索限制角速度
- 在 `main.js` 的 `updateGame()` 中，旋转前先预测，碰撞后限制 omega
- **不要在物理碰撞响应中处理炮管旋转限制**，会导致逻辑混乱

### 5. 碰撞检测的多层采样
- `getAllCollisionCorners()` 返回车体角 + 炮管角 + 边缘中点 + 凹槽拐角
- 额外采样点填充车体前脸与炮管之间的凹槽，防止墙体穿入
- **添加新碰撞形状时必须更新采样点**

### 6. 硬约束安全网（`hardConstraintWalls`）
- 在物理迭代后执行，确保绝不穿墙
- 迭代 3 次处理多面墙交汇处
- 推出位置的同时消除穿入方向的速度分量
- **这是最后防线，不要依赖它来替代正常碰撞响应**

### 7. 触摸控制的设备检测逻辑
- 优先使用 User Agent 检测（最可靠）
- 桌面系统即使有触摸 API 也判定为非触摸设备
- 小屏幕 + 触摸支持 → 触摸设备
- **不要简单用 `'ontouchstart' in window` 判断**，会误判触摸屏笔记本

### 8. 摇杆死区机制
- 死区内（`distance <= deadZone`）：只旋转不移动
- 死区外：旋转 + 移动
- 角度始终更新，距离和强度在死区内归零
- **修改摇杆逻辑时必须保持这个行为**

### 9. 胜利缓冲期系统
- 只剩一人时启动 `VICTORY_GRACE_TIME`（2.5秒）缓冲期
- 记录 `pendingWinnerId`，缓冲期结束后重新检测存活状态
- 如果最后存活者也死了 → 平局
- **不要在检测到只剩一人时立即判定胜利**

### 10. 坐标转换链
屏幕坐标 → Canvas 物理坐标 → 游戏逻辑坐标：
```javascript
canvasX = (clientX - rect.left) * (canvas.width / rect.width)
gameX = (canvasX - VIEWPORT_OFFSET_X) / VIEWPORT_SCALE
```
**所有触摸/鼠标事件必须经过这个转换**

## 代码风格
- 使用 ES6 类（`class Tank`），但不使用模块系统
- 函数式辅助工具（`utils.js` 中的 `vecAdd`、`vecRotate` 等）
- 物理引擎中的内部函数使用 `_` 前缀（如 `_applyWallImpulse`）
- 常量使用 `UPPER_SNAKE_CASE`
- 类方法使用 `camelCase`

## 常见陷阱
1. **不要在 `constants.js` 中使用 `const` 声明常量** - 会导致其他文件无法访问
2. **不要修改坦克速度时忘记更新 `_lastInputVx/Vy`** - 会破坏碰撞残余提取
3. **不要在碰撞响应中直接设置 `tank.angle`** - 应该通过 `omega` 和 `integrateVelocity()` 更新
4. **不要跳过 `fitCanvas()` 中的视口变换计算** - 触摸控制依赖这些全局变量
5. **不要在 AI 代码中假设 `tank.input` 会被保留** - 每帧开始时会被重置
